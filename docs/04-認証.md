# 認証（メール＋パスワード）

「メールアドレスとパスワードで登録 → メール認証 → ログイン」を、**この技術構成（Cloudflare Workers + D1 + Vercel）**に合わせて、セキュアに実装できます。

## 仕組み（ざっくり）

```
[ブラウザ] --登録/ログイン--> [Cloudflare Worker(API)] --保存--> [D1(データベース)]
                                     |
                                     +--確認メール--> [Resend] --> ユーザーのメール
```

- **登録**: ユーザーを「未確認」で作成し、確認メールを送る
- **メール認証**: メールのリンクを開くと確認完了 → ログイン画面に戻る
- **ログイン**: 確認済みのユーザーだけログインでき、セッション Cookie が発行される

## セキュリティのポイント

| 項目 | 実装 |
|------|------|
| パスワード | PBKDF2-HMAC-SHA256（21万回・ソルト付き）でハッシュ化。平文は保存しない |
| パスワード照合 | 定数時間比較（タイミング攻撃対策） |
| セッション | D1 セッション。Cookie には生の値、DB には SHA-256 を保存。サーバー側で即失効可能 |
| Cookie | `HttpOnly; Secure; SameSite=None`（フロントとバックが別ドメインのため） |
| 確認トークン | ランダム32バイト、24時間で失効、使い切り。DB にはハッシュのみ |
| CORS | 認証 Cookie を通すため `Allow-Credentials: true` ＋ 許可オリジンを厳密指定 |
| 情報漏洩対策 | 登録・ログインの失敗応答を一律化し、メールアドレスの存在を悟らせない |

## ハーネスでの有効化

`harness.config.json` の `auth.mode` で挙動が決まります。

| mode | 挙動 |
|------|------|
| `"ask"` | setup 中にまず **「認証（ログイン）機能は必要ですか？」** と確認する（推奨） |
| `"on"` | 必ず有効化（雛形配置＋D1＋シークレット＋スキーマ適用） |
| `"off"` | 何もしない |

`ask` の流れ:

1. **雛形を作る前に**「この SaaS に認証（ログイン）機能は必要ですか？」と質問する
   （フロントに既存のログイン画面が無ければ既定 Yes、有れば既定 No）
2. **不要**と答えたら、何もせず次へ（設定に `off` を保存し、次回は聞き直さない）
3. **必要**と答えたとき:
   - 既存のログイン画面が**無い** → そのまま雛形を実装
   - 既存のログイン画面が**有る** → 「用意された雛形も追加しますか？」と再確認（既定 No・既存ファイルは上書きしません）

「実装する」を選ぶと、ハーネスが自動で次を行います。

1. 認証の雛形コードを配置
   - バックエンド: `＜backend.dir＞/auth/`（API）と `＜backend.dir＞/migrations/0001_auth.sql`（テーブル定義）
   - フロント（Next.js の場合）: `＜frontend.dir＞/app/login`・`app/register`・`lib/auth-client.ts`
2. D1（データベース）を有効化
3. 必要なシークレットをヒアリング（`RESEND_API_KEY`・`EMAIL_FROM`）
4. D1 にスキーマ（テーブル）を適用（`cloudflare-migrate` ステップ）
5. フロントのオリジンを CORS 許可（`wire-up` ステップ）

一度決めた選択は `harness.config.json` に保存され、次回以降は聞き直しません。

## 事前に用意するもの

- **Resend アカウント**（無料枠あり）: <https://resend.com>
  - API Keys → Create API Key で `RESEND_API_KEY` を取得
  - 送信元にするドメイン/アドレスを「検証（verify）」しておく（未検証だと送信できません）
  - 検証済みアドレスを `EMAIL_FROM` に設定（例: `no-reply@あなたのドメイン`）

## 実装後にやること（アプリへの組み込み）

配置された `＜backend.dir＞/auth/README.md` に沿って、認証ルートをアプリにマウントします（Hono の例つき）。
最小構成では、バックエンドのエントリで `registerAuthRoutes(app)` を呼ぶだけです。

```ts
import { Hono } from "hono";
import { registerAuthRoutes, type AuthBindings } from "./auth/routes";

const app = new Hono<{ Bindings: AuthBindings }>();
registerAuthRoutes(app);
export default app;
```

依存に `hono` が必要です（`npm i hono`）。

## 動作確認

デプロイ後、`saas-launch doctor` で以下が確認できます。

- `RESEND_API_KEY` / `EMAIL_FROM` が設定済みか（未設定なら ❌ と直し方を表示）
- バックエンド到達性・CORS 連携

画面の流れ:

1. `https://＜あなたのフロント＞/register` で登録 → 確認メールが届く
2. メールのリンクを開く → `/login?verified=1` に戻る
3. `/login` でログイン → セッション発行

## よくある質問

- **メールが届かない**: `EMAIL_FROM` が Resend で検証済みか、`RESEND_API_KEY` が正しいか確認。
- **ログインできない（未確認エラー）**: 先にメールのリンクを開いて確認を完了してください。
- **ブラウザで CORS エラー**: `saas-launch doctor` で CORS 連携を診断。`wire-up` を再実行すれば直ります。
- **セッションが保持されない**: フロントの API 呼び出しが `credentials: "include"` になっているか確認（雛形は対応済み）。
