# セキュリティ

このリポジトリは「第三者が自分のローカルアプリを Cloudflare + Vercel にデプロイするための
ハーネス（CLI）＋認証テンプレート」です。ここでは、その前提でのセキュリティ設計と、
利用者が守るべきことをまとめます。

## ワンコマンドインストーラ（install.sh / install.ps1）の設計

- 書き込み先は `~/.saas-launch`（Windows は `%USERPROFILE%\.saas-launch`）内のみ。
  **管理者権限（sudo）は要求せず**、システム全体の設定・既存の Node.js 環境には触れません。
- Node.js は **nodejs.org 公式配布物のみ**を取得し、公式の `SHASUMS256.txt` に対する
  **SHA-256 チェックサム検証**を通過したものだけを展開します。
- ツール本体は HTTPS（github.com / codeload.github.com）から取得します。
  取得元は環境変数 `SAAS_LAUNCH_REPO` / `SAAS_LAUNCH_BRANCH` で明示的に上書きしない限り固定です。
- 環境に影響する操作（シェル設定ファイル・ユーザー PATH への1行追加）は、
  **実行前に必ずユーザーの同意**を求めます。同意しない場合もフルパス実行で利用できます。
- インストーラは冪等で、再実行するとツールを最新版へ更新します。

## ハーネス本体（CLI）の設計

### シークレットの取り扱い

- APIキー等のシークレットは**マスク入力**でヒアリングし、`wrangler secret put` /
  `vercel env add` に **stdin 経由**で渡します。コマンドライン引数には載せません
  （`ps` やシェル履歴に残さないため）。
- シークレットの**値はどのファイルにも保存しません**。状態ファイル
  `.saas-launch-state.json` には「登録が完了したシークレット名」だけを記録し、
  再開時の二重登録を防ぎます。URL などの秘密でない値のみ `collected` に保存します。
- 状態ファイルは所有者のみ読み書き可（パーミッション 0600）で保存し、
  `.gitignore` によりコミット対象から除外しています。セットアップ成功時に自動削除されます。
- `generate: true` のシークレット（自動生成キー）は生成時に**一度だけ画面に表示**します。
  必要ならその場でパスワードマネージャ等に控えてください（ファイルには残りません）。

### コマンド実行

- 外部コマンドは **シェルを介さず**（`execa` の argv 直接指定）実行します。
- ユーザー入力・設定ファイル由来の値（シークレット名・DB名・パス等）は
  **配列形式の argv** として渡すため、値が別の引数やフラグに化けることはありません。
- さらに `harness.config.json` のスキーマ（zod）で、シークレット名・環境変数名は
  `[A-Za-z_][A-Za-z0-9_]*`、プロジェクト名・DB名は英数字始まりの安全な文字集合に制限しています。
- 例外は `backend.deployCommand` で、これは**利用者自身が自分のマシンで実行するデプロイ
  コマンドを宣言する**ための項目です。第三者から受け取った `harness.config.json` を
  そのまま使う場合は、実行前に `deployCommand` の内容を必ず確認してください。

## 認証テンプレート（templates/auth）の設計

デプロイ後にインターネットへ公開されるのはこの部分です。既定で次の対策が入っています。

| 脅威 | 対策 |
|------|------|
| パスワード漏洩 | PBKDF2-HMAC-SHA256（210,000回・16バイトソルト）。平文は保存しない |
| タイミング攻撃 | 照合は定数時間比較。登録時は既存ユーザーでも同コストのハッシュ計算を実行 |
| ブルートフォース | D1 固定ウィンドウのレート制限（ログイン: IP毎 30回/15分＋アカウント毎 10回/15分） |
| メール爆撃 | 登録エンドポイントを IP毎 10回/時に制限 |
| CPU 枯渇（DoS） | パスワード長を 8〜200 文字に制限。保存ハッシュの反復回数も 10万〜500万回に制限 |
| セッション乗っ取り | セッションIDは Cookie にのみ生値、DB には SHA-256。`HttpOnly; Secure` Cookie |
| CSRF | 状態を変えるメソッドは Origin ヘッダを検証（`SameSite=None` 前提のため必須） |
| CORS の誤設定 | ワイルドカード不使用。許可オリジンを環境変数の許可リストと厳密照合して echo |
| ユーザー列挙 | 登録・ログインの失敗応答を一律化 |
| トークン漏洩 | メール確認トークンは 256bit ランダム・24時間失効・使い切り。DB にはハッシュのみ |
| キャッシュ汚染 | 認証応答はすべて `Cache-Control: no-store` |

レート制限のしきい値は、コピーされた `auth/routes.ts` 冒頭の定数で調整できます。

## 利用者が守ること

1. **`harness.config.json` と `.saas-launch-state.json` をコミットしない**
   （`.gitignore` 済みですが、対象アプリ側のリポジトリでも同様に除外してください）。
2. **`.dev.vars` / `.env` を対象アプリでコミットしない**（wrangler のローカルシークレット）。
3. 第三者から受け取った `harness.config.json` は、`deployCommand` を確認してから実行する。
4. Resend / Stripe 等の APIキーは、本番用と開発用を分け、漏洩時は即座に失効・再発行する。
5. 認証テンプレートを改変する場合、`README.md`（テンプレート内）のセキュリティ設計を
   崩していないか確認する。特に Cookie 属性・Origin 検証・レート制限は外さないこと。

## 脆弱性の報告
